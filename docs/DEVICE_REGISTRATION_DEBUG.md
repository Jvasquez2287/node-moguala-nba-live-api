# Device Registration Troubleshooting Guide

## Issue: "Sent notifications to 0/1 users" or Device Registration Not Working

When you see "`[Expo] Sent notifications to 0/1 users`" in the logs, it means the system tried to send notifications but couldn't find any registered devices for the user. This guide will help you diagnose the problem.

## Prerequisites

You need:
- Valid JWT authorization token
- Your Clerk ID (usually available from your authentication provider)
- An Expo push token (generated by your mobile app)

## Step 1: Verify User Registration

First, check if your user account exists in the database and validate your credentials.

### Endpoint: Check User Info
```bash
curl -X GET http://localhost:8000/api/v1/notifications/debug/check-user \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "x-clerk-id: YOUR_CLERK_ID"
```

**Headers needed:**
- `Authorization: Bearer YOUR_JWT_TOKEN` - Your JWT token with "Bearer " prefix
- `x-clerk-id: YOUR_CLERK_ID` - Your Clerk ID from authentication

**Expected Response (Success):**
```json
{
  "success": true,
  "user": {
    "id": 1,
    "clerk_id": "user_123abc",
    "email": "user@example.com",
    "first_name": "John",
    "last_name": "Doe"
  },
  "validationResult": {
    "valid": true,
    "user": {
      "id": 1,
      "clerk_id": "user_123abc",
      "email": "user@example.com",
      "first_name": "John",
      "last_name": "Doe",
      "stripe_id": "cus_xxx"
    }
  }
}
```

**If this fails:**
- Make sure your JWT token is correct
- Make sure your x-clerk-id header matches your actual Clerk ID
- Make sure you're using "Bearer " prefix before your token
- Check that your user was created in the database

## Step 2: Register a Device Token

Register your mobile device's push notification token.

### Endpoint: Register Device
```bash
curl -X POST http://localhost:8000/api/v1/notifications/register-device \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "x-clerk-id: YOUR_CLERK_ID" \
  -H "Content-Type: application/json" \
  -d '{
    "token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx]",
    "deviceName": "My iPhone",
    "osType": "ios"
  }'
```

**Request body:**
- `token` *(required)* - Your Expo push token (starts with `ExponentPushToken[`)
- `deviceName` *(optional)* - Friendly name for your device
- `osType` *(optional)* - Device OS: `ios`, `android`, or `web`

**Expected Response (Success):**
```json
{
  "success": true,
  "message": "Device token registered successfully",
  "userId": "user_123abc"
}
```

**If this fails:**
- Check token format - must start with `ExponentPushToken[`
- Check that token is valid and hasn't changed
- Check server logs for error details

---

## Step 3: Verify Device Registration

After registering a device, verify it was actually stored in the database.

### Endpoint: Check Registered Tokens
```bash
curl -X GET http://localhost:8000/api/v1/notifications/debug/check-tokens \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "x-clerk-id: YOUR_CLERK_ID"
```

**Expected Response (Success):**
```json
{
  "success": true,
  "userId": "user_123abc",
  "tokenCount": 1,
  "tokens": [
    {
      "token": "ExponentPushToken[xxxxxxxxxxxxxxxxx...",
      "device_name": "My iPhone",
      "os_type": "ios",
      "is_active": 1,
      "created_at": "2026-02-12T19:30:00.000Z",
      "last_used": "2026-02-12T19:30:00.000Z"
    }
  ]
}
```

**If `tokenCount` is 0:**
- The device registration endpoint returned success, but the token wasn't stored
- This indicates a database connection issue
- Check server logs for SQL errors
- Verify the `device_tokens` table exists
- Verify foreign key constraint (user must exist in `users` table)

## Step 4: Check Server Logs

The enhanced logging will show you exactly what's happening:

### Look for these log messages:

**Successful registration:**
```
[Notification Route] Registering device token for user xxx: { token, deviceName, osType }
[Expo] registerDeviceToken called with userId: user_123 token: ExponentPushToken[...
[Expo] Token format is valid, checking if token already exists...
[Expo] Token is new, inserting into database...
[Expo] Device token registered for user: user_123. Insert result: {...}
[Expo] Verification - tokens found for user user_123: 1
```

**Failed registration:**
```
[Notification Route] Invalid token validation
[Notification Route] No clerk_id found in validation result
[Expo] userId is empty or undefined
[Expo] Invalid push token format: xxx
```

## Common Issues and Solutions

### Issue 1: "No clerk_id found in validation result"
**Cause:** The `x-clerk-id` header is missing or invalid

**Solution:**
```bash
# Make sure to include this header:
curl ... -H "x-clerk-id: YOUR_ACTUAL_CLERK_ID"
```

### Issue 2: "Device token registered successfully" but tokens don't appear
**Cause:** Database connection issue or foreign key constraint violation

**Solution:**
1. Check that user exists in database:
   ```bash
   curl ... http://localhost:8000/api/v1/notifications/debug/check-user
   ```
2. Check server logs for SQL errors
3. Verify `device_tokens` table has foreign key to `users.clerk_id`

### Issue 3: "Invalid push token format"
**Cause:** Token doesn't start with `ExponentPushToken[` or is malformed

**Solution:**
1. Regenerate token from your Expo app
2. Make sure you're copying the entire token
3. Don't add any spaces or extra characters

### Issue 4: "User not found"
**Cause:** The clerk_id doesn't match any user in the database

**Solution:**
1. Make sure you created an account on the app
2. Get the correct clerk_id from your authentication provider
3. Make sure you're using the same clerk_id consistently

## Database Troubleshooting

### Check if device_tokens table exists:
```sql
SELECT * FROM INFORMATION_SCHEMA.TABLES 
WHERE TABLE_NAME = 'device_tokens'
```

### Check registered devices for a user:
```sql
SELECT * FROM device_tokens 
WHERE user_id = 'user_123abc'
ORDER BY last_used DESC
```

### Verify foreign key relationship:
```sql
SELECT *
FROM INFORMATION_SCHEMA.KEY_COLUMN_USAGE
WHERE TABLE_NAME = 'device_tokens' 
  AND COLUMN_NAME = 'user_id'
```

## Complete Workflow Example

Here's a complete example of registering a device and checking it:

```bash
# 1. Check user exists
curl -X GET http://localhost:8000/api/v1/notifications/debug/check-user \
  -H "Authorization: Bearer eyJhbG..." \
  -H "x-clerk-id: user_123abc"

# 2. Register device token
curl -X POST http://localhost:8000/api/v1/notifications/register-device \
  -H "Authorization: Bearer eyJhbG..." \
  -H "x-clerk-id: user_123abc" \
  -H "Content-Type: application/json" \
  -d '{
    "token": "ExponentPushToken[abcd1234...]",
    "deviceName": "iPhone 14",
    "osType": "ios"
  }'

# 3. Verify token was registered
curl -X GET http://localhost:8000/api/v1/notifications/debug/check-tokens \
  -H "Authorization: Bearer eyJhbG..." \
  -H "x-clerk-id: user_123abc"

# 4. Check registered devices
curl -X GET http://localhost:8000/api/v1/notifications/devices \
  -H "Authorization: Bearer eyJhbG..." \
  -H "x-clerk-id: user_123abc"
```

## Getting Your Necessary Information

### Getting your JWT Token:
- This is provided by your authentication system (Clerk)
- Include it with "Bearer " prefix in Authorization header

### Getting your Clerk ID:
- In Clerk dashboard: https://dashboard.clerk.com/
- In your app: Usually in localStorage or user object
- Example format: `user_2a47Tx6C0uXXXXXXXXXXXXXX`

### Getting your Expo Push Token:
- From your React Native/Expo app using:
  ```javascript
  import * as Notifications from 'expo-notifications';
  const token = (await Notifications.getExpoPushTokenAsync()).data;
  ```
- Should look like: `ExponentPushToken[abcd1234...]`

## Need More Help?

If you're still having issues:

1. **Check the server console** - All actions are logged with `[Expo]`, `[Notification Route]`, or `[TokenCheck]` prefixes
2. **Enable detailed query logging** - Check database query results
3. **Verify all headers are correct** - Authorization token, Clerk ID, content-type
4. **Test with a simple curl command** first before integrating into your app
5. **Check database directly** for any errors or missing data
